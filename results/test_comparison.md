# MARVEL 测试结果对比：Hungarian 联合分配 vs 原始冲突解决

## 测试配置

| 参数 | 值 |
|------|-----|
| 测试地图集 | maps_test |
| 测试数量 | 100 episodes |
| 智能体数量 | 6 |
| 视场角 (FoV) | 120° |
| 传感器范围 | 10m |
| 最大步数 | 128 |
| 策略模式 | Greedy (确定性) |

## 关键发现

### 1. 成功率断崖式下跌（99% → 44%）

关闭 Hungarian 后，超过一半的 episode 无法在 128 步内完成探索。根本原因：6 个机器人的原始冲突解决机制（距离排序 + 最近邻空间替换）在 Agent 数量增加时严重退化——机器人倾向于互相跟踪，导致大范围感知重叠。

### 2. 重叠率飙升（24.2% → 91.5%）

这是所有指标中变化最剧烈的。91.5% 意味着 6 个机器人几乎完全扎堆在同一区域，只有 8.5% 的感知区域是互不重叠的。原始冲突解决的"事后修补"策略在 6 Agent 场景下完全失效。

### 3. 行进距离翻倍（312.6m → 606.4m）

因为重叠率极高，机器人需要走更多路才能覆盖相同面积。配合成功率下降，说明即便走到 128 步上限，大量 episode 仍然未能完成探索。

### 4. 稳定性变差

- Std 行进距离从 111m 增至 182m (+63%)
- Std 到 90% 距离从 55m 增至 126m (+128%)

表明不同地图下的表现差异显著放大，系统鲁棒性下降。

## Hungarian 联合分配效果总结

| 维度 | 改善幅度 |
|------|---------|
| 探索率 | +13.5% |
| 成功率 | +125% (55pp) |
| 行进效率 | −48.5% (少走一半路) |
| 协调性（重叠率） | −73.6% (重叠减少近 3/4) |
| 稳定性（Std 行进） | −38.7% |
| 时间效率 | −48.5% |

---

## 结论

在 6 Agent 的多机器人探索场景中，Hungarian 全局联合分配**不是锦上添花的优化，而是系统正常工作的必要条件**。原始的独立采样 + 事后冲突修补方案在 Agent 数量 ≥ 4 时出现了严重的协调退化问题，匈牙利算法以可忽略的计算开销（< 1ms/步）换取了决定性的性能提升。
